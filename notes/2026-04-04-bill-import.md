# 2026-04-04 微信 / 支付宝账单导入需求变动说明

## 一、改动背景

当前产品已经具备基础记账能力，但用户在从微信、支付宝迁移历史账单时，仍然需要手动逐条补录，操作成本很高，且非常容易放弃。对于已有大量消费记录的用户来说，“能否快速导入历史账单”已经不是加分项，而是决定是否继续使用产品的关键能力。

在第一轮导入能力落地后，我们又在真实使用中发现，导入问题并不只是“能否上传文件”这么简单，而是包含了完整的导入体验链路：

- 文件能不能被正确读取
- 账单格式能不能被识别
- 哪些记录应该导入、哪些应该跳过
- 为什么跳过、为什么失败，用户能不能看懂
- 系统判断错了之后，用户有没有核对和理解空间

因此，这次需求变动的本质，不是单点补 bug，而是把“账单导入”从一个技术能力，升级成一个可以真正被用户理解和信任的产品流程。

## 二、改动目的

本次改动主要服务于两个产品目标：

1. 降低历史账单迁移门槛，让新用户或重度用户可以快速把微信、支付宝流水带进 App。
2. 提升导入结果的可理解性和可控性，让用户知道“系统为什么这么判断”，而不是只能看到“导不进来”。

更具体地说，我们希望解决以下用户问题：

- 用户上传真实文件后没有结果，不知道是文件坏了、格式不支持，还是系统识别失败
- 用户看到部分记录没导入，但分不清是重复、失败，还是被系统主动过滤
- 用户无法逐条核对系统判断，容易对导入结果失去信任
- 系统把真实消费错判成“中性流水”，会直接破坏用户对账本准确性的认知

## 三、改动目标

从产品角度，这次需求变动的目标不是“支持更多文件类型”这么简单，而是完成一条闭环体验：

- 支持用户导入支付宝真实导出的 `CSV` 和微信真实导出的 `.xlsx`
- 在导入前完成文件识别、内容解析、记录标准化、分类、去重、去噪和异常归因
- 让用户在真正入库前，就能看到完整预览结果
- 把“重复项”“已跳过”“失败项”明确区分开，而不是混成“没导进来”
- 让用户可以逐条展开核对，理解每条记录的系统判断依据
- 保证真实消费、真实收入、退款尽可能保留，减少对统计结果有误导性的噪音流水

## 四、需求变动说明

### 1. 入口与首屏流程调整

为了减少导入前的认知负担，导入流程改成“先选账户，再选文件”的顺序。这样用户在一开始就知道这批账单将归入哪个账户，也减少了导入完成后再补选账户的反复操作。

对应策略：

- 在个人页新增“导入账单”入口
- 新建独立导入页，不把导入流程塞进新增账单页
- 首屏去掉大段说明文案，保留核心操作：
  - 页面标题
  - 账户选择
  - 文件上传按钮

### 2. 真实文件兼容能力补齐

产品目标不再停留在“理论支持微信和支付宝”，而是要兼容用户真实导出的账单文件。

已识别出的真实差异包括：

- 支付宝账单常见为 `GBK/GB18030` 编码，而不是 UTF-8
- 微信账单常见为 `.xlsx`
- 微信 `.xlsx` 表头并不在第一行，前面会有说明性内容

因此这次把“文件接入层”改成了真实场景优先：

- 支付宝 `CSV` 先读二进制，再尝试不同编码解码，并根据表头命中率挑选最佳结果
- 微信 `.xlsx` 支持跳过说明行，识别真实表头后再解析
- 无法识别的文件给出明确失败原因，不再让用户只看到“没有结果”

### 3. 导入预览从“结果提示”升级为“核对界面”

原始导入预览更像一个静态结果页，只告诉用户成功了多少、失败了多少，但不支持深入理解。实际使用中，用户真正关心的是：

- 哪些会导入
- 哪些不会导入
- 为什么
- 我能不能逐条检查

因此本次把预览页定义成“导入前核对界面”。

预览需要明确分成四类：

- 将要导入
- 重复项
- 已跳过
- 失败项

并且每条记录都支持展开查看：

- 系统判断
- 原始交易时间
- 交易对方
- 商品说明 / 商品
- 收支
- 金额
- 支付方式
- 平台交易号

同时，为了适配长列表核对场景，`将要导入`、`已跳过`、`失败项` 还增加了整组折叠 / 展开能力，并用上下箭头明确表示当前状态。

### 4. 去重逻辑从“弱规则”升级为“平台交易号优先”

账单导入里最容易伤害用户信任的问题之一，是把两笔真实不同的交易误判为重复。原始方案主要依赖日期、金额、说明做弱指纹，这在一些同日同金额的小额交易场景下容易误伤。

本次需求变动后，去重逻辑改成：

- 优先使用平台交易号作为强指纹
- 缺失时才回退到时间、金额、说明、付款方式等组合特征

这样做的产品价值是：

- 降低同一天多笔相似交易被误杀的概率
- 提高重复识别的可解释性
- 让用户更容易接受“为什么这条会被判重复”

### 5. 去噪规则从“平台流水视角”改成“日常记账视角”

支付平台账单是全量资金流水，但记账产品不一定需要把所有流水都原样搬进来。用户真正关心的是“我的日常消费、收入、退款”，而不是理财系统内部每一笔资金移动。

因此本次对“已跳过”逻辑做了产品化取舍：

默认跳过的记录类型：

- `不计收支`
- 余额宝收益发放
- 余额宝自动转入
- 明确的账户充值 / 提现 / 转存
- 交易关闭等无效记录

默认保留的记录类型：

- 真实消费支出
- 真实收入
- 退款记录

在真实使用中，我们又进一步修正了两个判断边界：

- `余额宝支付` 本身不是中性流水，不能因为支付方式是余额宝就跳过
- `校园卡充值` 不应被简单归入“中性流水”，因为它更接近一笔真实消费支出
- `分账-基础软件服务费` 作为真实费用支出，也不再默认跳过

这意味着我们不再从“支付平台如何记账”的角度定义噪音，而是从“用户是否希望它进入个人账本”的角度来定义。

## 五、改动成果

本次需求变动落地后，导入能力已经从“初步可用”提升到“具备真实使用价值”：

- 新增独立导入页，并完成与个人页的入口打通
- 支持支付宝真实 `CSV` 和微信真实 `.xlsx` 的识别与解析
- 支持导入前预览、账户归属选择、批量导入与导入后数据刷新
- 预览页可清楚展示“将要导入 / 重复项 / 已跳过 / 失败项”
- 预览页支持整组折叠与逐条展开核对
- 去重逻辑升级为平台交易号优先
- 去噪逻辑更加贴近日常记账场景，减少真实消费被误跳过
- 用户现在可以看到系统判断依据，显著提升对导入结果的信任感

## 六、成果收益

从产品价值上看，这次需求变动带来了几类直接收益：

### 1. 用户迁移成本下降

用户可以更快把微信、支付宝历史流水带入 App，减少从“尝试使用”到“形成留存”的阻力。

### 2. 导入可理解性增强

用户不再只看到“导不进来”，而是可以理解具体原因。这个变化对用户信任非常关键。

### 3. 数据准确性提升

真实消费、真实收入、退款被更稳定地保留下来，噪音流水被更合理地过滤，首页和统计页的结果更符合用户预期。

### 4. 后续迭代基础更稳

当前已经形成了较清晰的导入架构：文件读取、模板识别、标准化、去重、去噪、预览、批量导入。后续扩展更多平台或更复杂的导入规则时，不需要推倒重做。

## 七、结果说明

从本次实现结果来看，当前版本已经满足以下预期：

- 支付宝真实导出文件不会再因为编码问题直接失败
- 微信 `.xlsx` 可以被正确识别并进入预览
- 用户在导入前能看到完整分类结果
- `将要导入`、`已跳过`、`失败项` 可以折叠和展开
- 每条记录都可以逐条展开核对
- `余额宝支付` 的正常消费不会再误判为中性流水
- `校园卡充值` 和 `平台服务费` 已调整为默认导入

## 八、当前限制

- 当前仅兼容支付宝 `CSV` 和微信 `.xlsx`
- 暂不支持用户自定义去噪规则和分类映射
- 未命中分类规则的记录仍统一落到“其他”
- 还未系统化兼容更多银行、记账平台或更多微信 / 支付宝模板变体

## 九、后续建议

- 增加用户可配置的“保留 / 跳过规则”，让导入策略更个性化
- 支持在预览页对单条记录做“本次保留 / 本次跳过”的人工干预
- 持久化平台交易号，进一步增强跨多次导入的稳定去重能力
- 扩展更多账单来源，逐步把导入能力做成产品的核心迁移工具
